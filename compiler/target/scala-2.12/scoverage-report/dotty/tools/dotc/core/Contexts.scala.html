51   /** A context is passed basically everywhere in dotc.
52    *  This is convenient but carries the risk of captured contexts in
53    *  objects that turn into space leaks. To combat this risk, here are some
54    *  conventions to follow:
55    *
56    *    - Never let an implicit context be an argument of a class whose instances
57    *      live longer than the context.
58    *    - Classes that need contexts for their initialization take an explicit parameter
59    *      named `initctx`. They pass initctx to all positions where it is needed
60    *      (and these positions should all be part of the intialization sequence of the class).
61    *    - Classes that need contexts that survive initialization are instead passed
62    *      a "condensed context", typically named `cctx` (or they create one). Condensed contexts
63    *      just add some basic information to the context base without the
64    *      risk of capturing complete trees.
65    *    - To make sure these rules are kept, it would be good to do a sanity
66    *      check using bytecode inspection with javap or scalap: Keep track
67    *      of all class fields of type context; allow them only in whitelisted
68    *      classes (which should be short-lived).

316     /** The context for a supercall. This context is used for elaborating
317      *  the parents of a class and their arguments.
318      *  The context is computed from the current class context. It has
319      *
320      *  - as owner: The primary constructor of the class
321      *  - as outer context: The context enclosing the class context
322      *  - as scope: The parameter accessors in the class context
323      *  - with additional mode: InSuperCall
324      *
325      *  The reasons for this peculiar choice of attributes are as follows:
326      *
327      *  - The constructor must be the owner, because that's where any local methods or closures
328      *    should go.
329      *  - The context may not see any class members (inherited or defined), and should
330      *    instead see definitions defined in the outer context which might be shadowed by
331      *    such class members. That's why the outer context must be the outer context of the class.
332      *  - At the same time the context should see the parameter accessors of the current class,
333      *    that's why they get added to the local scope. An alternative would have been to have the
334      *    context see the constructor parameters instead, but then we'd need a final substitution step
335      *    from constructor parameters to class parameter accessors.
336      */

338       val locals = newScopeWith(owner.typeParams ++ owner.asClass.paramAccessors: _*)
